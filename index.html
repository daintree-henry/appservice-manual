App Service

  • Azure's Platform-as-a-Service (PaaS) for deploying and running web applications
  • Provides automatic scaling, deployment, and monitoring without managing servers or infrastructure

Creating an App Service

First, go to the Azure Portal, search for App Services, and click on it.

Click the "Create" button, and you'll be prompted to choose one of the following four app types.

App Service Creation Options

Here is a breakdown of each available option:

  • Web App
    • Provides a hosting environment for dynamic web applications.
    • Example use cases: feature-rich applications, API servers.
    • Recommended when the app includes backend logic.
      ※ Check with your developer to confirm if a backend is included.
  • Static Web App
    • Hosts static websites built with HTML, CSS, and JavaScript.
    • Example use cases: landing pages, product info pages without login functionality.
    • Recommended for frontend-only projects.
      ※ Confirm with the frontend developer whether this option is suitable.
  • Web App + Database
    • Creates a Web App along with a database (e.g., SQL) in one step.
    • Example use cases: applications requiring a new database such as membership systems or forums.
    • If an existing database is already in place, use the regular Web App option. Choose this if a new DB setup is needed.
  • WordPress on App Service
    • Offers a ready-to-use WordPress environment.
    • Example use cases: internal blogs, marketing websites.
    • Use this option only if WordPress is specifically required.

  • Click Create, and then select Web App.

Zone redundancy

Zone redundancy is a high-availability feature that distributes your App Service instances across multiple Availability Zones (AZs), ensuring continued service even in the event of a zone failure.
This feature is only available with Premium v2 or higher pricing tiers and cannot be changed after deployment.

Sample Workflow for Python 3.12

Once connected, the following file will be committed automatically to:
simpleapp/.github/workflows/main_simpleapp.yml

# Docs for the Azure Web Apps Deploy action: https://github.com/Azure/webapps-deploy
# More GitHub Actions for Azure: https://github.com/Azure/actions
# More info on Python, GitHub Actions, and Azure App Service: https://aka.ms/python-webapps-actions

name: Build and deploy Python app to Azure Web App - simpleapp

# Trigger workflow on push to main branch
on:
  push:
    branches:
      - main
  workflow_dispatch:

jobs:
  build:
	  # Use the latest Ubuntu runner provided by GitHub
    runs-on: ubuntu-latest
    permissions:
      contents: read 
      
    steps:
	    # Check out source code from the GitHub repository
      - uses: actions/checkout@v4

			# Set up Python 3.12 environment
      - name: Set up Python version
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'
          
      # Create and activate virtual environment
      - name: Create and start virtual environment
        run: |
          python -m venv venv
          source venv/bin/activate
      
      # Install required dependencies
      - name: Install dependencies
        run: pip install -r requirements.txt
        
      # Zip the entire application into release.zip
      - name: Zip artifact for deployment
        run: zip release.zip ./* -r

			# Upload release.zip as an artifact for the deploy job
      - name: Upload artifact for deployment jobs
        uses: actions/upload-artifact@v4
        with:
          name: python-app
          path: |
            release.zip
            !venv/

  deploy:
    runs-on: ubuntu-latest
    needs: build # Run this job only after build is complete
    environment:
      name: 'Production'
      url: ${{ steps.deploy-to-webapp.outputs.webapp-url }}
    permissions:
      id-token: write # Required for Azure OIDC login
      contents: read

    steps:
	    # Download release.zip from the build job
      - name: Download artifact from build job
        uses: actions/download-artifact@v4
        with:
          name: python-app
          
			# Unzip the deployment package
      - name: Unzip artifact for deployment
        run: unzip release.zip

      # Authenticate with Azure using OIDC
      - name: Login to Azure
        uses: azure/login@v2
        with:
          client-id: ${{ secrets.__clientidsecretname__ }}
          tenant-id: ${{ secrets.__tenantidsecretname__ }}
          subscription-id: ${{ secrets.__subscriptionidsecretname__ }}
	
			# Deploy the app to Azure Web App
      - name: 'Deploy to Azure Web App'
        uses: azure/webapps-deploy@v3
        id: deploy-to-webapp
        with:
          app-name: 'simpleapp'
          slot-name: 'Production'

GitHub Actions requires deployment permissions to Azure.
By default, this is handled using OIDC (OpenID Connect) with the following Azure credentials:

          client-id: ${{ secrets.__clientidsecretname__ }}
          tenant-id: ${{ secrets.__tenantidsecretname__ }}
          subscription-id: ${{ secrets.__subscriptionidsecretname__ }

These values must be configured as Secrets in your GitHub repository.

Networking

Configure the network settings for your application.

Enable public access

This setting determines whether your application is accessible from the public internet. When On, the application is accessible via the App Service URL specified in the Basics tab. When Off, the app is only accessible from within a private network.

By default, App Service instances are not integrated into a Virtual Network (VNet).
To allow communication with private resources (e.g., a database), Virtual Network Integration must be enabled.

If you configured a database in the Database tab, Virtual Network Integration will be automatically enabled to ensure connectivity.
In this case, the option is locked and cannot be changed manually.

💡 Note: If no database is created, the default value for Virtual Network Integration is Off.

Inbound access

This option becomes available when Virtual Network Integration is enabled.
It allows you to create Private Endpoints for App Service. App Service cannot have both Public access and Private Endpoints enabled at the same time. If Public access is set to On, the Private Endpoints option is locked as Off.

Outbound access

When Virtual Network Integration is enabled, you can also enable Outbound access. This lets you route outbound traffic from the App Service through a specific virtual network (VNet), allowing greater control over external connections.

Inbound access와 Outbound access 비교

CategoryInbound accessOutbound access
FunctionControls incoming traffic to the App ServiceControls outgoing traffic from the App Service
PurposeExternal users accessing the appApp accessing internal DBs, APIs, etc.

Application Setup

Now, go to the App Service and click on the Default domain to open the application in your browser.

You will see that the application fails to load.

Click Log stream to view the live logs from the running App Service instance.

2025-05-05T04:51:38.633333Z RuntimeError: DB_USER, DB_PASSWORD, DB_HOST, DB_PORT, DB_NAME 환경변수를 모두 설정해주세요.

Applications like simpleapp are designed to retrieve database credentials from environment variables at runtime, since these settings differ by environment.
Below is an excerpt from the code that loads those variables:

# app.py
# Load configuration from environment variables
db_user = os.getenv("DB_USER")
db_password = quote_plus(os.getenv("DB_PASSWORD", ""))
db_host = os.getenv("DB_HOST")
db_port = os.getenv("DB_PORT", "5432")
db_name = os.getenv("DB_NAME", "todo_db")

The os.getenv("ENV_NAME") function reads the value of an environment variable on the server.
For example,
os.getenv("DB_NAME", "todo_db") uses "todo_db" as a default value if DB_NAME is not set. To resolve the error, you must define at least these three environment variables: DB_USER, DB_PASSWORD, and DB_HOST.

In the App Service, go to Settings > Configuration > Application settings to define environment variables.

Switch to the Connection strings tab and locate the database connection string that was automatically created when the PostgreSQL server was provisioned.

Click Show value to reveal the credentials, then copy them. Note that the format is designed for ASP.NET, and therefore not directly usable in Python applications. You’ll need to extract the necessary values and define them individually as environment variables under the Application settings tab.

Click Add, and enter the following keys and values (example shown):

# Value copied from AZURE_POSTGRESQL_CONNECTIONSTRING
Database=simpleapp-database;Server=simpleapp-server.postgres.database.azure.com;User Id=tahflkqcxe;Password=DbT...

# Set these as environment variables:
DB_HOST=simpleapp-server.postgres.database.azure.com
DB_USER=tahflkqcxe
DB_PASSWORD=DbT...

After adding the environment variables, click Apply to save the changes.

Once saved, return to Log stream and confirm that the error has disappeared. If everything is configured correctly, reloading the App Service URL should now display the working application.

Bug Report

1. Error When Creating PostgreSQL and Redis Together:

Issue:

When both services attempt to use Private Endpoints, the error PrivateEndpointCreationNotAllowedAsSubnetIsDelegated is triggered.
This happens because the subnet is already delegated to PostgreSQL, and Redis cannot create a Private Endpoint on a subnet that has already been delegated to a different service.

Error Details:

{
  "code": "PrivateEndpointCreationNotAllowedAsSubnetIsDelegated",
  "message": "Private endpoint /subscriptions/a91b0e13-a6ce-40ab-b044-f951da8f9327/resourceGroups/appsvc-rg/providers/Microsoft.Network/privateEndpoints/simpleapp-redis-privateEndpoint cannot be created as subnet /subscriptions/a91b0e13-a6ce-40ab-b044-f951da8f9327/resourceGroups/appsvc-rg/providers/Microsoft.Network/virtualNetworks/vnet-siqrzogi/subnets/subnet-eyrdumoyscobk is delegated.",
  "details": []
}

Error Screen:

Diagnosis:

When the Virtual Network is created, two subnets are automatically provisioned:

  • 10.0.1.0/24: Subnet delegated to Microsoft.Web/serverfarms, used exclusively for App Service outbound traffic
  • 10.0.2.0/24: Subnet delegated to Microsoft.DbforPostgreSQL/flexibleServers, used exclusively for the PostgreSQL Flexible Server

A single delegation per subnet is supported.

The issue occurs because the Redis Cache resource attempts to create a Private Endpoint in a subnet that is already delegated to PostgreSQL, which results in the error:

PrivateEndpointCreationNotAllowedAsSubnetIsDelegated

3. Auto-Generate App Settings for PostgreSQL Integration

Currently, the AZURE_POSTGRESQL_CONNECTIONSTRING generated during PostgreSQL integration is provided in a format optimized for ASP.NET environments. However, it cannot be used directly in Python applications. Developers must manually parse the string and re-register key values such as DB_HOST, DB_USER, and DB_PASSWORD as separate environment variables. This process is repetitive and prone to human error.

From the perspective of App Service users, having the platform handle such setup tasks automatically would significantly improve usability.

Suggestion: If the key components of the connection string could be automatically extracted and registered separately in App Settings during PostgreSQL integration, it would greatly enhance the overall user experience.
