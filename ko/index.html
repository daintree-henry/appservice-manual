App Service

  • 서버나 인프라 관리 없이 자동 확장, 배포, 모니터링 기능까지 제공

App Service 생성

먼저 Azure Portal에서 App Services를 검색하고, 클릭합니다.

"Create" 버튼을 클릭하면 다음 4가지 App 유형 중 하나를 선택할 수 있습니다.

App Service의 생성 옵션

아래는 각 옵션에 대한 세부 설명입니다.

  • Web App
    • 동적인 웹 애플리케이션의 호스팅 환경을 제공합니다.
    • 예시: 특정 기능을 제공하는 애플리케이션, API 서버 등
    • 백엔드가 포함된 경우 Web App 배포를 권장합니다.
      ※ 백엔드 여부는 개발자에게 확인하세요.
  • Static Web App
    • HTML, CSS, JavaScript 기반의 정적 웹사이트를 호스팅합니다.
    • 예시: 로그인 기능이 없는 소개 페이지, 제품 랜딩 페이지 등
    • 프런트엔드 단독 프로젝트인 경우 Static Web App 사용을 권장합니다.
      ※ 사용 가능 여부는 프론트엔드 개발자와 사전에 협의하세요.
  • Web App + Database
    • Web App과 데이터베이스(SQL 등)를 함께 생성합니다.
    • 예시: 신규로 DB까지 필요한 회원 시스템, 게시판 등
    • 기존 데이터베이스가 있는 경우에는 일반 Web App을 선택하고, 새로운 데이터베이스 구성이 필요한 경우 이 옵션을 선택합니다.
  • WordPress on App Service
    • WordPress를 손쉽게 설치하고 운영할 수 있는 환경입니다.
    • 예시: 사내 블로그, 마케팅용 콘텐츠 웹사이트 등
    • WordPress 사용이 필수인 경우에만 선택하세요.

  • Create를 클릭하고, Web App을 선택합니다.

Basics

리소스의 기본적인 정보를 설정합니다.

Project Details

  • Subscription/Resource Group

    리소스를 생성할 구독과 리소스 그룹을 지정합니다.

Instance Details

  • Name

    리소스의 이름을 지정합니다.

    리소스 생성 시 아래 형식으로 접속 URL이 생성됩니다. 따라서 리소스 명은 고유해야 합니다.

    (리소스명).azurewebsites.net

    Secure unique default hostname 옵션을 켜면, 난수와 지역명이 포함된 고유하고 보안이 강화된 URL이 생성됩니다.

    (리소스명)-(난수)-(지역명).azurewebsites.net

  • Name 필드에 고유한 리소스 명을 지정합니다.
  • ‘Secure unique default hostname on’ 옵션을 활성화합니다.

  • Publish

    배포 형태를 지정합니다.

    Code: 소스 코드를 배포합니다.

    Container: 컨테이너 이미지를 배포합니다.

Code vs Container 방식 비교

항목CodeContainer
설명Web App의 사전 준비된 환경에서 코드 빌드 및 실행실행 환경이 포함된 컨테이너 이미지를 그대로 실행
장점설정이 간단하여 빠른 배포 가능, Azure 플랫폼에 최적화된 환경 제공환경 제어 가능, 로컬과 동일한 조건 유지 가능
단점기본 실행 환경에 개발환경을 맞춰야 함Dockerfile 작성, 이미지 빌드 등 구성 복잡도 높음

일반적인 웹 앱에는 Code 방식을 사용하고, 실행 환경 설정이 복잡하거나 특정 의존성이 필요한 경우에는 Container 방식을 권장합니다.

  • Publish: Code를 선택합니다.

  • Runtime stack

    Code 옵션 선택 시에만 활성화되며, 코드를 실행할 프로그래밍 언어 환경을 설정할 수 있습니다.

    .NET, JAVA, Node, PHP, Python 환경을 지원합니다.

  • Operating System

    실행 환경의 OS를 선택합니다.

  • Region

    실행할 리소스의 지역을 선택합니다.

  • Runtime stack: Python 3.12을 선택합니다. Python은 OS로 Linux만 사용할 수 있습니다.
  • Region: Korea Central을 선택합니다.

Zone redundancy

Zone redundancy는 App Service 인스턴스를 여러 가용 영역(AZ)에 분산시켜 장애 발생 시에도 서비스가 중단되지 않도록 보장하는 고가용성 기능입니다. 이 기능은 Premium v2 요금제 이상에서 사용 가능하며, 설정 후 변경할 수 없습니다.

  • Zone redundancy 옵션을 확인합니다. Basic B1 플랜을 선택했기 때문에 Disabled로 고정됩니다.

Database

App Service 생성 시 데이터베이스 리소스를 함께 생성할 수 있습니다. 함께 생성하면 네트워크 및 리소스 그룹 구성이 자동으로 이루어져 편리하지만, 세부적인 설정을 직접 조정할 수 없습니다.따라서 개발·테스트 환경처럼 빠른 구성이 필요한 경우에 적합합니다.

아래와 같이 데이터베이스 서버 정보를 입력합니다.

  • Create a Database 옵션을 선택합니다.
  • Engine: PostgreSQL - Flexible Server
  • Server name: simpleapp-server
  • Database name: simpleapp-database

위 설정에 따라 Web App과 함께 PostgreSQL 데이터베이스 서버가 자동으로 생성됩니다.

  • Deployment 탭으로 이동합니다.

Continuous deployment settings

Continuous deployment를 활성화하면, 소스 코드 변경 시 자동으로 App Service에 배포할 수 있습니다.

Azure Web App은 GitHub Actions 외에도 ZIP 파일 업로드, FTP, 로컬 Git, Azure Pipelines 등을 포함한 다양한 배포 방식을 제공합니다.

관련문서
  • Continuous deployment: Enable를 선택합니다.

Github settings

깃허브와 연동하여 배포 파이프라인을 구성할 수 있습니다.

GitHub account

GitHub 계정 연동을 위해 Authorize application 팝업에서 로그인 및 권한 허용을 진행합니다.

  • 자신의 깃허브 계정을 연동합니다.

연동이 완료되면 Organization, Repository, Branch 항목이 자동으로 표시됩니다.

  • Repository: Fork한 simpleapp을 선택합니다.
  • Branch: main을 선택합니다.

Workflow configuration

Github Actions와 연동하려면 소스코드에 워크플로우 구성 파일이 자동으로 추가됩니다. Preview file 클릭 시 App Service 환경에 필요한 워크플로우 파일을 확인할 수 있습니다.

Python 3.12 워크플로우 파일 예시

GitHub Actions 연동 시, 아래와 같은 워크플로우 파일이 simpleapp/.github/workflows/main_simpleapp.yml 경로에 자동으로 커밋됩니다.

# Docs for the Azure Web Apps Deploy action: https://github.com/Azure/webapps-deploy
# More GitHub Actions for Azure: https://github.com/Azure/actions
# More info on Python, GitHub Actions, and Azure App Service: https://aka.ms/python-webapps-actions

name: Build and deploy Python app to Azure Web App - simpleapp

# main 브랜치에 push 시 동작하는 워크플로우
on:
  push:
    branches:
      - main
  workflow_dispatch:

jobs:
  build:
	  # GitHub에서 제공하는 최신 Ubuntu 실행 환경 사용
    runs-on: ubuntu-latest
    permissions:
      contents: read 
      
    steps:
	    # GitHub 리포지토리에서 소스코드 체크아웃
      - uses: actions/checkout@v4

			# Python 3.12 환경 설정
      - name: Set up Python version
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'
          
      # 가상 환경 생성 및 활성화
      - name: Create and start virtual environment
        run: |
          python -m venv venv
          source venv/bin/activate
      
      # 의존성 설치
      - name: Install dependencies
        run: pip install -r requirements.txt
        
      # 앱 전체를 압축하여 release.zip 파일 생성
      - name: Zip artifact for deployment
        run: zip release.zip ./* -r

			# 다음 단계(deploy)에서 사용할 수 있도록 release.zip을 아티팩트로 업로드
      - name: Upload artifact for deployment jobs
        uses: actions/upload-artifact@v4
        with:
          name: python-app
          path: |
            release.zip
            !venv/

  deploy:
    runs-on: ubuntu-latest
    needs: build # build job이 완료되어야 실행됨
    environment:
      name: 'Production'
      url: ${{ steps.deploy-to-webapp.outputs.webapp-url }}
    permissions:
      id-token: write # Azure 로그인에 필요한 권한
      contents: read

    steps:
	    # 이전 build 단계에서 업로드한 release.zip 아티팩트 다운로드
      - name: Download artifact from build job
        uses: actions/download-artifact@v4
        with:
          name: python-app
          
			# release.zip 파일 압축 해제
      - name: Unzip artifact for deployment
        run: unzip release.zip

      # Azure 로그인 - OIDC(토큰 기반 인증) 사용
      - name: Login to Azure
        uses: azure/login@v2
        with:
          client-id: ${{ secrets.__clientidsecretname__ }}
          tenant-id: ${{ secrets.__tenantidsecretname__ }}
          subscription-id: ${{ secrets.__subscriptionidsecretname__ }}
	
			# Azure Web App으로 앱 배포
      - name: 'Deploy to Azure Web App'
        uses: azure/webapps-deploy@v3
        id: deploy-to-webapp
        with:
          app-name: 'simpleapp'
          slot-name: 'Production'

GitHub Actions에서 실행되는 배포 서버는 Azure에 배포할 수 있는 권한이 필요합니다. 기본 구성은 Azure의 clinetId, tenantId, subscriptionId를 사용한 OICD 인증을 사용합니다. 아래 세 값은 GitHub Actions의 Secret으로 지정해야 합니다.

          client-id: ${{ secrets.__clientidsecretname__ }}
          tenant-id: ${{ secrets.__tenantidsecretname__ }}
          subscription-id: ${{ secrets.__subscriptionidsecretname__ }

Authentication settings

아래 Basic authentication 옵션을 사용하면 OIDC 없이도, 사용자 이름과 비밀번호만으로 소스 코드 배포를 허용하는 설정입니다. GitHub와 연동하면 App Service가 자동으로 배포 파이프라인을 구성하므로, OIDC 또는 Basic 인증 중 어느 방식을 사용해도 설정이 완료됩니다.

  • Basic authentication: Disable을 선택합니다.

Basic authentication 옵션을 활성화한 상태에서 GitHub Actions를 연동하면, 아래와 같은 워크플로우 파일이 simpleapp/.github/workflows/main_simpleapp.yml 경로에 자동으로 커밋됩니다.

# More GitHub Actions for Azure: https://github.com/Azure/actions
# More info on Python, GitHub Actions, and Azure App Service: https://aka.ms/python-webapps-actions

name: Build and deploy Python app to Azure Web App - simpleapp

# main 브랜치에 push 시 동작하는 워크플로우
on:
  push:
    branches:
      - main
  workflow_dispatch:

jobs:
  build:
	  # GitHub에서 제공하는 최신 Ubuntu 실행 환경 사용
    runs-on: ubuntu-latest
    permissions:
      contents: read 
      
    steps:
	    # GitHub 리포지토리에서 소스코드 체크아웃

			# Python 3.12 환경 설정

        uses: actions/setup-python@v5
        with:
          python-version: '3.12'
          
      # 가상 환경 생성 및 활성화
      - name: Create and start virtual environment
        run: |
          python -m venv venv
          source venv/bin/activate
      
      # 의존성 설치
      - name: Install dependencies
        run: pip install -r requirements.txt
        
      # 앱 전체를 압축하여 release.zip 파일 생성
      - name: Zip artifact for deployment
        run: zip release.zip ./* -r

			# 다음 단계(deploy)에서 사용할 수 있도록 release.zip을 아티팩트로 업로드
      - name: Upload artifact for deployment jobs
        uses: actions/upload-artifact@v4
        with:
          name: python-app
          path: |
            release.zip
            !venv/

  deploy:
    runs-on: ubuntu-latest
    needs: build # build job이 완료되어야 실행됨
    environment:
      name: 'Production'
      url: ${{ steps.deploy-to-webapp.outputs.webapp-url }}
    permissions:
      id-token: write # Azure 로그인에 필요한 권한
      contents: read

    steps:
	    # 이전 build 단계에서 업로드한 release.zip 아티팩트 다운로드
      - name: Download artifact from build job
        uses: actions/download-artifact@v4
        with:
          name: python-app
          
			# release.zip 파일 압축 해제
      - name: Unzip artifact for deployment
        run: unzip release.zip

      # Azure Web App으로 앱 배포 (Basic authentication - Publish Profile 방식)
      - name: Deploy to Azure Web App
        uses: azure/webapps-deploy@v2
        with:
          app-name: 'simpleapp'  # 실제 App Service 이름
          publish-profile: ${{ secrets.AZURE_WEBAPP_PUBLISH_PROFILE }}
          package: release.zip

이 경우 GitHub Actions의 Secret으로 인증 정보가 자동으로 연동됩니다. Github > Settings > Security > Secrets and variables > Actions > Repository secrets 에서 자동 생성된 인증 정보를 확인할 수 있습니다.

Networking

애플리케이션의 네트워크 구성을 설정할 수 있습니다.

Enable public access

애플리케이션이 외부에서 접근 가능한지 설정합니다. On으로 설정하면, Basics 탭에서 지정한 App Service URL을 통해 외부에서 접속할 수 있습니다. Off로 설정할 경우 내부망에서만 접근이 가능합니다.

기본적으로 App Service의 인스턴스는 Virtual Network에 통합되지 않는 리소스이기 때문에, 데이터베이스와 같은 Private Network 리소스와 통신하려면 Virtual Network Integration을 On으로 설정해야 합니다. Database 탭에서 데이터베이스 서버를 함께 구성한 경우, App Service는 내부 리소스와의 통신을 위해 Virtual Network Integration을 자동으로 활성화합니다.
이 경우 해당 옵션은 고정되며 수동으로 변경할 수 없습니다.

💡 참고: Database를 생성하지 않은 경우, Virtual Network Integration의 기본 값은 Off입니다.

Inbound access

Virtual network integration을 On으로 설정할 경우 활성화되는 옵션입니다. App Service에 접속할 수 있는 Private endpoints를 생성할 수 있습니다. App Service는 Public access와 Private endpoints 옵션을 동시에 활성화할 수 없습니다. Public access가 On인 경우 Private Endpoints 옵션은 Off로 고정됩니다.

Outbound access

Virtual network integration을 On으로 설정할 경우 Outbound access 옵션이 활성화됩니다. 이 옵션을 활성화하면, App Service에서 실행 중인 애플리케이션에서 외부로 나가는 트래픽의 경로를 특정 VNet으로 지정하고 제어할 수 있습니다.

Log stream을 클릭하면, 애플리케이션이 실행 중인 서버(인스턴스)의 로그를 확인할 수 있습니다.

2025-05-05T04:51:38.633333Z RuntimeError: DB_USER, DB_PASSWORD, DB_HOST, DB_PORT, DB_NAME 환경변수를 모두 설정해주세요.

이 에러는 데이터베이스 접속 정보를 담은 환경 변수가 설정되지 않아 발생한 것입니다.
simpleapp과 같은 백엔드 애플리케이션은 환경마다 접속 정보가 다르기 때문에, 실행 시점에 환경 변수에서 해당 정보를 불러오도록 설계되어 있습니다. 아래 코드는 환경 변수를 불러오는 일부 코드입니다.

# app.py
# Load configuration from environment variables
db_user = os.getenv("DB_USER")
db_password = quote_plus(os.getenv("DB_PASSWORD", ""))
db_host = os.getenv("DB_HOST")
db_port = os.getenv("DB_PORT", "5432")
db_name = os.getenv("DB_NAME", "todo_db")

os.getenv(”환경변수명”) 코드는 실행 서버의 환경 변수를 읽어옵니다. os.getenv("DB_NAME", "todo_db")는 환경 변수 DB_NAME이 설정되지 않은 경우 "todo_db"를 기본값으로 사용합니다. DB_USER, DB_PASSWORD, DB_HOST 세 개의 변수를 지정해야 합니다.

Settings > Environment variables를 클릭하면, 현재 애플리케이션 서버의 환경 변수를 지정할 수 있습니다.

Connection strings 탭을 클릭하면, Web App과 함께 생성된 PostgreSQL 데이터베이스 서버의 접속 정보가 자동으로 저장되어 있는 것을 확인할 수 있습니다.

Show value를 선택해 접속 정보를 확인하고, 복사합니다. Connecitons strings에 자동 생성된 정보는 기본적으로 ASP.NET 전용 형식으로 설정되어 있어 Python에서는 직접 사용할 수 없습니다. 따라서, 해당 정보를 복사해 App Settings 탭에 개별 환경 변수로 설정해주어야 합니다.

App Setting 탭을 클릭한 후, Add 버튼을 눌러 새로운 환경 변수를 지정합니다. 아래의 예시 값을 참고합니다.

# AZURE_POSTGRESQL_CONNECTIONSTRING에 저장되어 있는 값
Database=simpleapp-database;Server=simpleapp-server.postgres.database.azure.com;User Id=tahflkqcxe;Password=DbT...

# 생성해야 하는 환경 변수
DB_HOST=simpleapp-server.postgres.database.azure.com
DB_USER=tahflkqcxe
DB_PASSWORD=DbT...

세 개의 환경 변수가 추가된 것을 확인한 후, Apply를 클릭하여 저장합니다.

설정 후 다시 Log stream에서 에러가 사라졌는지 확인한 뒤, App Service의 URL로 접속하면 애플리케이션이 정상적으로 동작하는 것을 확인할 수 있습니다.

3. PostgreSQL 통합을 위한 App Setting 자동 생성

현재 PostgreSQL 연동 시 자동으로 생성되는 AZURE_POSTGRESQL_CONNECTIONSTRING은 ASP.NET 환경에 최적화된 형식으로 제공되며, Python 애플리케이션에서는 바로 사용할 수 없어 개발자가 수동으로 파싱 후 DB_HOST, DB_USER, DB_PASSWORD 등으로 재등록해야 합니다. 이 과정은 반복적이고 실수가 발생하기 쉬운 작업입니다.

App Service를 사용하는 고객 입장에서는, 플랫폼이 설정을 자동화해주는 방식이 훨씬 더 편리하게 느껴질 수 있습니다.

제안: PostgreSQL 연동 시 주요 키 값을 자동으로 App Settings에 분리 등록해주는 기능이 추가된다면, 사용자 경험이 보다 향상될 것으로 기대됩니다.
